* Hardware — FXS / ATA configuration

VoIP gateway / ATA setup notes for the analogue phones connected to
this PBX. The =[ata]= endpoint template in =pjsip.conf= covers
SIP-to-FXS adapters: plug a regular analogue phone into the adapter's
=FXS Phone= jack (RJ-11), the adapter handles the SIP side.

Extension range for ATAs is =6010-6019= (per =pjsip.conf=). Each FXS
port on a multi-port adapter registers as its own extension.

** Common settings

Identical across all three devices (Grandstream HT812, Cisco SPA122,
Cisco ATA191-MPP). Set these first, then move on to the per-device
specifics below.

| Setting              | Value                                |
|----------------------+--------------------------------------|
| SIP server / proxy   | =$PBX= (the Asterisk host)           |
| SIP port             | =5060=                               |
| Transport            | UDP                                  |
| Outbound proxy       | (leave blank)                        |
| User / Auth ID       | =6010=, =6011=, … (one per FXS port) |
| Password             | matching =pjsip.conf=                |
| Register expires     | 300 s (5 min)                        |
| Preferred codecs     | G.711 µ-law (=PCMU=) then A-law (=PCMA=) |
| DTMF mode            | RFC2833                              |
| NAT traversal        | off (LAN — set =Keep-Alive= if remote) |
| Use DNS SRV          | no                                   |

Our =[ata]= template restricts the negotiated codec list to ulaw/alaw,
so picking anything else on the adapter just shortens the SDP offer
and isn't useful.

** Which Ethernet port to use

The HT812 has two RJ45 jacks because it ships as a small NAT router:

- =WAN= — uplink toward your network. Gets an IP by DHCP (or falls
  back to =192.168.2.1= if no lease).
- =LAN= — downstream port. By default the device hands out DHCP and
  NATs traffic from anything plugged in here.

For this PBX setup the HT812 is *just a SIP endpoint*, not a router.
So:

- *Plug the cable into the =WAN= port.* That's the one labelled
  "INTERNET" on the silkscreen.
- *Leave the =LAN= port unplugged.*
- After first boot, log into the web UI and set
  =Basic Settings → Device Mode= to *Bridge* (sometimes called
  "Network Mode"). This disables the built-in DHCP server and NAT so
  the unit really is just an ATA — no router behaviour to surprise you
  later.

** Finding it on the network

Grandstream's OUI is =00:0b:82=. Every HT81x's MAC starts with it,
which makes "find me a Grandstream on this LAN" much easier than
guessing IPs. Pick the lowest-friction path that works for your case
below.

*** 1. Check the neighbour cache first (zero tools, no sudo)

If anyone on this host has talked to the HT81x recently — even just an
ICMPv6 router-solicit — Linux's neighbour cache already has its MAC:

#+begin_src bash
  ip neigh | rg -i '00:0b:82'
#+end_src

A hit means the device is alive on that interface. =REACHABLE= is
fresh, =STALE= means it was seen earlier and might still be there
(re-ping to verify), =FAILED= means it's gone.

*** 2. Active sweep when it has an IPv4 lease

If the HT81x got a DHCP lease (normal case), sweeping the subnet
populates the ARP table and you grep the OUI back out. Three flavours
depending on what's installed:

#+begin_src bash
  # Best when arp-scan is installed (uses raw ARP, no ICMP needed)
  sudo arp-scan --interface=<iface> --localnet | rg '00:0b:82'

  # nmap ping-sweep — also needs sudo to send ARP
  sudo nmap -sn -e <iface> 192.168.1.0/24
  ip neigh show dev <iface> | rg '00:0b:82'

  # No-sudo fallback: TCP-probe :80 across the subnet. Each handshake
  # forces an ARP resolve, so the neighbour cache fills up naturally.
  for i in $(seq 1 254); do
    (nc -z -w1 192.168.1.$i 80 >/dev/null 2>&1 && echo "192.168.1.$i:80") &
  done; wait
  ip neigh show dev <iface> | rg '00:0b:82'

  # Passive sniffing — handy if the device isn't talking yet
  sudo netdiscover -S -f -i <iface>
#+end_src

The IPv4 next to the matching MAC is the one to browse to.

*** 3. Factory-default fallback at =192.168.2.1=

With no DHCP lease the HT81x hardcodes its WAN to =192.168.2.1=. That
IP isn't on your normal subnet (=192.168.1.0/24=), so ICMP/TCP from
your host won't even leave the box — the kernel has no route to it.
Add a temporary in-subnet alias on the interface and the device shows
up at the link layer:

#+begin_src bash
  sudo ip addr add 192.168.2.10/24 dev <iface>
  # Probe :80 across the factory-default range
  for i in $(seq 1 254); do
    (nc -z -w1 192.168.2.$i 80 >/dev/null 2>&1 && echo "192.168.2.$i:80") &
  done; wait
  ip neigh show dev <iface> | rg '00:0b:82'
  # Cleanup once you've moved it to DHCP:
  sudo ip addr del 192.168.2.10/24 dev <iface>
#+end_src

The "right" fix after this is to log into the web UI and set
=Basic Settings → IP Address → Address Type= to =DHCP=, then reboot —
the device will then take a lease in your normal subnet and the alias
is no longer needed.

*** 4. IPv6 link-local — works even without any IPv4

The HT81x replies on IPv6 link-local regardless of its IPv4 state, so
you can open its web UI without faffing with alias subnets at all.
EUI-64 derives the v6 address straight from the MAC: flip bit 1 of the
first octet and insert =ff:fe= in the middle.

For MAC =00:0b:82:XX:YY:ZZ= → =fe80::20b:82ff:feXX:YYZZ=.

Probe and ping:

#+begin_src bash
  # Force the v6 ping out the wired interface
  ping6 -c1 -I <iface> fe80::20b:82ff:feb7:be38
  # Then check the neighbour cache for any 00:0b:82
  ip -6 neigh show dev <iface> | rg '00:0b:82'
#+end_src

Open the web UI directly (URL-encode the =%= before the zone-id):

#+begin_example
  http://[fe80::20b:82ff:feb7:be38%25<iface>]/
#+end_example

Firefox tip: pasting that into the address bar gets it routed to the
search engine instead of opened. Workarounds: right-click → "Paste &
Go", press Alt+Enter, or put the URL inside an =<a href="...">= in a
local HTML file and click it.

Once you're in, =Status → Network Status= shows the current IPv4 (if
any) and the firmware version.

Press and hold the rear =RESET= button for ~7 seconds. The manual
specifies the Ethernet cable should be unplugged while doing this.

** SIP configuration

Web UI at =http://<device-ip>=, log in as =admin=. The HT812 has a
per-port config page (=FXS PORT1= and =FXS PORT2=); fields below apply
to each port separately:

| HT812 field                | Value                          |
|----------------------------+--------------------------------|
| Account Active             | Yes                            |
| Primary SIP Server         | =$PBX=                         |
| SIP Transport              | UDP                            |
| SIP User ID                | =6010= (port 1), =6011= (port 2) |
| Authenticate ID            | same as SIP User ID            |
| Authenticate Password      | matching =pjsip.conf=          |
| Name (display)             | e.g. "Hallway phone"           |
| Preferred Vocoder choice 1 | PCMU                           |
| Preferred Vocoder choice 2 | PCMA                           |
| Send DTMF                  | via RTP (RFC2833)              |
| NAT Traversal              | No (LAN)                       |
| Register Expiration        | 5 (minutes)                    |
| Use Random SIP Port        | No                             |

After saving, the =Status= page should show =Registered= for the port
you configured. Cross-check on the PBX:

#+begin_src bash
  docker exec -it $(docker ps -qf ancestor=sm6wjm/asterisk) \
      asterisk -rx "pjsip show endpoint 6010"
#+end_src

The contact line should list the HT812's LAN IP.

** Firmware upgrade

Grandstream hosts current and beta firmware on a public HTTP server.
Process that's known to work:

1. Web UI → =Maintenance → Upgrade and Provisioning=.
2. Set =Always Check for New Firmware= to enabled.
3. Set =Firmware Upgrade Via= to HTTP.
4. =Firmware Server Path= → =firmware.grandstream.com=.
5. =Save and Apply=.
6. Reboot the device and let the upgrade finish.
